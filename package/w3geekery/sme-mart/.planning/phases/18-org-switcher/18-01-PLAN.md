# Plan 18.1: Org Switcher Service & User Menu

**Phase:** 18 — Org Switcher
**Requirements:** OS-01, OS-02, OS-03, OS-04 (partial), OS-05
**Effort:** 4–6 hours
**Depends on:** None

## Goal

Enable users to switch between their organizations via a nested mat-menu submenu in the user profile dropdown. Integrate ZeroBias SDK's `selectOrg()` method with client-side defensive filtering (hidden, System Org, ops-org placeholder). Trigger a hard-reload post-switch to drop all cached app state and load new org context.

After this plan, users can:
1. Click "Switch Organization" in the user menu
2. See an alphabetically-sorted list of orgs with the current org marked in bold with a leading dot
3. Click any org to trigger a blocking spinner dialog and org switch
4. Reload happens automatically; subsequent API calls use the new `dana-org-id` header

## Risks

1. **Dialog component doesn't exist or doesn't fit use case** — If `ConfirmDialogComponent` doesn't support a spinner-only mode (`disableClose: true`, no buttons), Task 1 will stall. Mitigation: Create a minimal custom spinner dialog component (20 lines, `src/app/shared/dialogs/switching-org-dialog/`) as a fallback. **Action in plan:** Verify ConfirmDialogComponent path and inputs before coding Task 1; if not suitable, build minimal custom dialog in sub-task.

2. **Hard reload clears app state too aggressively** — If some user-facing feature requires gradual state transition rather than full reload, users may see a brief flash or lose unsaved form state. Mitigation: This is acceptable per director guidance (matches ZB portal); if optimization needed, router refresh + explicit `CatalogService.clear()` call can be attempted post-Phase-18 UAT. **Action:** Document this as a known trade-off in PLAN summary.

3. **Ops-org filter undefined** — Without Kevin's clarification, `isOpsOrg(org)` is a no-op placeholder. If "ops orgs" leak into the menu on real orgs, users may see unexpected entries. Mitigation: TODO hook in service + defensive System Org + hidden filters still apply. Kevin will fill in the rule later. **Action:** Clear TODO comment in code; update BACKLOG with "ops-org filter definition" task for future phase.

4. **Currentorgs not pre-filtered server-side** — If `app.getOrgs()` doesn't pre-filter hidden/System/ops orgs, filtering only happens client-side. This is intentional per RESEARCH.md ("defensive filtering"). If server changes the filter behavior, client-side filters still protect. **Action:** None; this is by design.

5. **Submenu doesn't close after selection** — If `matMenuTriggerFor` submenu persists after click, user must manually close it. Mitigation: Add `[closeMenu]="true"` or manually call `orgMenu.closeMenu()` in `onSelectOrg()` callback. **Action:** Test in Task 3; if needed, add `if (this.orgMenu) this.orgMenu.closeMenu()` to handler.

### Architectural Notes

1. **No-op on same org:** `switchTo()` returns early if `org.id === currentOrgId()`. This avoids an unnecessary reload for accidental re-clicks of the current org. UI still allows click (no disabled state).

2. **Hard reload justification (per RESEARCH.md §Refresh Strategy):** Services like `CatalogService` cache global catalog data (roles, skills, frameworks) that don't reset on router navigation. A hard `window.location.reload()` is the safest approach to clear all in-memory caches. Matches ZB portal pattern.

3. **OrgSwitcherService eagerly loads orgs on construction:** Fine for MVP (org list is small, ~5–6 items). If performance becomes an issue, can defer to lazy loading on first submenu open (computed signal + subscription timing).

4. **Submenu max-height with scroll:** Set to ~400px to handle future expansions to 20+ orgs (if demo grows). Scroll is native Material behavior.
