# Plan 26-04 — Correct Fictional Class IDs in PipelineWriteService

## Objective

Replace two fictional class ID constants in `pipeline-write.service.ts` with the canonical platform-assigned UUIDs. The two consts were computed as deterministic UUID v5 hashes from schema names, but the actual classes were registered with platform-assigned UUIDs that do NOT match the v5 derivations. Pipeline.receive returns "No such Class" for both fictional IDs.

**Background.** `seed-zb-provider.ts` already documents this discovery (line 9) and uses the canonical MPI id directly. Plan 26-02 validated `7bcf86a5-...` empirically. Today the canonical EngagementVettingItem id was confirmed via `platform.Class.getClass` on UAT.

**Impact prior to fix.** Silent production failures since Plan 041 (MPI) and Plan 063 (vetting). `vendor-profile.service.ts` (lines 149/204/232) and `vetting.service.ts` (lines 184/226/283/309) wrap pushEntity calls in `.catch(err => console.error(...))` — the failures land in the console, never surface to the user, never halt flow. Every MPI write and every vetting-item write through these services has been failing.

**Out of scope (do not expand).**
- Phase 20 fire-and-forget audit + remediation (Director planning separately)
- Refactoring vendor-profile.service or vetting.service write paths
- Any other entity types — Director audit confirmed remaining 21 class IDs match platform
