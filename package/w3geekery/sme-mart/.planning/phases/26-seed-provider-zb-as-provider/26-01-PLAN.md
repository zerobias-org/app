<task type="checkpoint:decision" gate="blocking">
  <decision>Platform-Provider Distinguisher Mechanism</decision>
  <context>
    Three options from RESEARCH.md § Platform-Provider Distinguisher Decision Tree (Phase 26 requires exactly ONE):

    **Option (a) — Hydra tag `sme-mart.provider.platform`:**
    - Create new `hydra.Tag` with name `sme-mart.provider.platform`, `scope: "org"` (or `"system"`), `ownerId: ZB_ORG_UUID`
    - Push as `Object.tag: [{ value: "<tag-uuid>" }]` on every ZB MPI record during seed
    - Enables GQL discovery: `MarketplaceProfileItem(tag: {value: ".eq.<tag-uuid>"}) { ... }` returns platform-providers uniformly
    - Generalizes to future platform-providers (same pattern for new providers)
    - Test-friendly: can mock/filter by tag in unit tests
    - Requires: one-time tag creation (idempotent search-first) + tag-uuid threaded through seed code
    - **RECOMMENDED** in RESEARCH.md for symmetry with Phase 25's validated Object.tag pattern

    **Option (b) — MPI `provider_type` section:**
    - Add new section `provider_type` with `data: "platform"` for each ZB record
    - Stays inside existing MPI class shape; no platform-tag plumbing
    - Simpler to code (no extra hydra call)
    - Downside: pollutes company_info catalog with a non-profile field; Phase 28 form must skip it; less discoverable
    - Acceptable but less elegant than (a)

    **Option (c) — Hardcoded orgId filter:**
    - In Browse Providers service, hardcode: `if (orgId === ZB_ORG_UUID) { /* platform provider */ }`
    - Zero data plumbing; trivial implementation
    - Downside: hardcoded UUID in code; doesn't generalize to future platform-providers; ugly to test; breaks on org-rename
    - Anti-pattern; avoid unless compelling reason

    **Research recommends (a)** for:
    - Alignment with Phase 25's validated Object.tag ingest pattern (used for W3Geekery Engagement + SmeMartProject)
    - Uniform GQL-level discovery matching marketplace tag patterns
    - Future-proofing: new platform-providers use same tag approach
    - Test coverage: tag filters are mockable; orgId hardcoding is not
  </context>
  <options>
    <option id="option-a">
      <name>Hydra tag `sme-mart.provider.platform`</name>
      <pros>
        - Symmetric with Phase 25's Object.tag mechanism (proven pattern)
        - Uniform GQL discovery: `MarketplaceProfileItem(tag: {value: ".eq.UUID"})` returns all platform-providers
        - Generalizes: future platform-providers use same tag UUID
        - Test-friendly: tag filters are mockable
        - Aligns with SME Mart's existing tag-based discovery patterns (engagements, projects)
      </pros>
      <cons>
        - Requires one-time hydra Tag.create call (but idempotent: search-first, create-only-if-missing)
        - Tag UUID must be threaded through seed code and tests
        - One extra field in MPI payload (tag array)
        - Hydra tag scope semantics unclear (org vs system) — may need Kevin clarification if org-level tag blocks discovery by other orgs
      </cons>
    </option>
    <option id="option-b">
      <name>MPI `provider_type` section with `data: "platform"`</name>
      <pros>
        - Stays entirely within MPI class shape (no platform-tag plumbing)
        - Simpler implementation: just add one more section to seed batch
        - No extra MCP call needed (tag creation)
      </pros>
      <cons>
        - Pollutes company_info catalog with a non-profile field
        - Phase 28 form schema must explicitly skip/hide the `provider_type` section (extra logic)
        - Less discoverable: would need code-side filter on section name, not uniform tag query
        - Doesn't generalize cleanly to future platform-providers (each would need its own section name variant)
        - Future maintainers may forget this section exists and try to surface it in the form
      </cons>
    </option>
    <option id="option-c">
      <name>Hardcoded `orgId == ZB_ORG_UUID` filter in Browse Providers</name>
      <pros>
        - Absolute simplest to code: one if-statement in provider-profiles.service
        - Zero data plumbing
      </pros>
      <cons>
        - Hardcoded UUID in application code (fragile; breaks if org-id ever changes)
        - Does NOT generalize: new platform-provider orgs would need code changes
        - Ugly to test: UUID mocking is brittle; tag mocking is clean
        - Anti-pattern per RESEARCH.md § Common Pitfalls "Hardcoded UUID for Platform Org in Browse Providers"
        - Creates maintenance debt: future devs will copy this pattern not knowing it's an anti-pattern
      </cons>
    </option>
  </options>
