## Installation & Configuration Details

### Hook Location & Pattern

**File:** `/Users/cstacer/Projects/w3geekery/zerobias-org-forks/app/.husky/pre-commit`  
**Size:** 3 lines (executable, v9 pattern)

**Content:**
```bash
#!/usr/bin/env sh
cd package/w3geekery/sme-mart
npx lint-staged
```

**Why monorepo-aware:** The git root is at `app/` level, but eslint and npm commands run from `sme-mart/` subdirectory. The hook changes directory before invoking lint-staged so it runs in the correct npm package context.

**V9 pattern notes:**
- No `. "$(dirname "$0")/_/husky.sh"` source line (v8 pattern; v9 doesn't create `_/husky.sh`)
- Direct `npx lint-staged` invocation (v9 simplification)
- Hook is executable and syntactically valid (verified with `sh -n`)

### lint-staged Configuration

**File:** `/Users/cstacer/Projects/w3geekery/zerobias-org-forks/app/package/w3geekery/sme-mart/.lintstagedrc.json`  
**Size:** 3 lines (valid JSON)

**Content:**
```json
{
  "*.{ts,html}": "eslint --max-warnings=0"
}
```

**Behavior:**
- Matches staged `.ts` and `.html` files
- Runs `eslint --max-warnings=0` on matched files
- `--max-warnings=0` enforces zero-warning policy (warnings fail the command, not just errors)
- Lint scope: staged files only, not the entire repo (keeps latency low)
- Non-matching file types (e.g., `.json`, `.md`, `.gitignore`) pass through without lint (as intended)

## Latency Measurement

**Scenario:** Stage 5 files and measure wall-clock commit time with the pre-commit hook.

**Test run 1 (cold cache, empty test files):**

**Result:** 3.914 seconds (cold cache)

**Test run 2 (warm cache, real content test files):**
```bash
cp src/app/app.component.ts src/app/test-perf-[1-5].component.ts
git add src/app/test-perf-*.component.ts
time git commit -m "perf test: hook latency with real files"
```
**Result:** 4.687 seconds (warm cache, real files)

**Analysis:**
- Both measurements exceed the 3-second target from the brief (brief note: "Hook latency above ~3s breaks Clark's commit cadence")
- Cold-start latency (3.9s) is just above the target; warm-start latency (4.7s) is significantly higher
- Latency is driven by:
  1. Node.js startup overhead (~1-2s)
  2. ESLint config loading + TypeScript parsing (eslint.config.js with flat config, tsconfig.json parsing)
  3. Plugin loading (@angular-eslint, typescript-eslint, @eslint/js)
  4. File processing (even empty files trigger the full pipeline)
- The MODULE_TYPELESS_PACKAGE_JSON warning in stderr adds overhead (eslint tries to parse as CommonJS, fails, reprses as ESM)

**Director question flagged:** Latency measurement exceeds target. See "Open Questions" section below.

After Plan 02's initial commit `76eaaef` landed, the Director surfaced a cross-package safety question and a latency optimization. Three additional atomic commits applied:

| Commit | File | Refinement |

| `fcabe35` | `app/.husky/pre-commit` | Early-exit gate: `git diff --cached --name-only \| grep -q '^package/w3geekery/sme-mart/'` — if no sme-mart files staged, exit 0 immediately. No Node startup penalty for sibling-package devs. |

### Cross-package safety mechanism (verifiable by sibling devs)

The hook lives at `app/.husky/pre-commit` (git root) and fires for ALL commits in this repo, including those touching sibling packages (`package/zerobias/data-explorer`, `package/zerobias/example-nextjs`, `package/zerobias/example-angular`). Without a gate, sibling commits would invoke `cd package/w3geekery/sme-mart && npx lint-staged` against staged files in OTHER packages — `lint-staged` would either no-op (no matches) or fail confusingly.

The early-exit gate is intentionally simple shell + grep — sibling-package devs reading the hook source can verify the boundary themselves without trusting `lint-staged`'s micromatch glob resolution. The first line of defense is "is this commit even touching sme-mart?" — answered in <100ms before any Node process spawns.
