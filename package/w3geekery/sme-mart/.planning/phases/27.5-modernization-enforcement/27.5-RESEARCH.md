## Rule Verification & Architecture Patterns

### Pattern 1: Standalone Components Only
**Verified rule name:** `@angular-eslint/prefer-standalone`  
**Plugin:** angular-eslint  
**Scope:** Component decorators  
**Configuration:**
```javascript
{
  "@angular-eslint/prefer-standalone": ["error"]
}
```
**What it bans:** Any component with explicit `standalone: false` property. Omitting the property or setting `standalone: true` is valid.  
**Status:** ✓ Verified. Rule exists and active in v21.3.1.

### Pattern 2: Field-Level `inject()` Over Constructor DI
**Verified rule name:** `@angular-eslint/prefer-inject`  
**Plugin:** angular-eslint  
**Scope:** Injectable classes, component constructors  
**Configuration:**
```javascript
{
  "@angular-eslint/prefer-inject": ["error"]
}
```
**What it bans:** Constructor parameters receiving injected dependencies. Examples: `constructor(private http: HttpClient)`, `constructor(@Inject(TOKEN) val)`, with metadata decorators like `@Optional()`.  
**Status:** ✓ Verified. Rule exists and active in v21.3.1.

### Pattern 3: Signal-based `input()` / `output()` Over `@Input()` / `@Output()` Decorators
**Verified rule name:** `@angular-eslint/prefer-signals` (for `@Input` + query decorators, NOT `@Output`)  
**Plugin:** angular-eslint  
**Scope:** Component class decorators  
**Configuration (partial — `@Input` only):**
```javascript
{
  "@angular-eslint/prefer-signals": [
    "error",
    {
      "preferInputSignals": true,
      "preferQuerySignals": true,
      "preferReadonlySignalProperties": true
    }
  ]
}
```
**What it bans:** `@Input()`, `@ViewChild()`, `@ViewChildren()`, `@ContentChild()`, `@ContentChildren()`. Signal replacements: `input()`, `viewChild()`, `viewChildren()`, `contentChild()`, `contentChildren()`.  
**Coverage gap — `@Output` decorators:** `prefer-signals` does NOT cover `@Output()` decorators. See Pattern 7 below for custom `no-restricted-syntax` rule.  
**Status:** ✓ Verified. Confirmed scope via v21.3.1 rule docs. Gap requires custom rule layer.

### Pattern 4: Control Flow Syntax `@if` / `@for` / `@switch`
**Verified rule name:** `@angular-eslint/template/prefer-control-flow`  
**Plugin:** @angular-eslint/eslint-plugin-template  
**Scope:** Template (`.html`) files  
**Configuration:**
```javascript
{
  "@angular-eslint/template/prefer-control-flow": ["error"]
}
```
**What it bans:** Structural directives `*ngIf`, `*ngFor`, `*ngSwitch` and property-binding variants `[ngIf]`, `[ngForOf]`, `[ngSwitch]`.  
**What it requires:** Built-in Angular v17+ syntax: `@if`, `@else if`, `@else`, `@for`, `@empty`, `@switch`, `@case`, `@default`.  
**Status:** ✓ Verified. Rule exists and active in v21.3.1.

### Pattern 5: Ban `import { CommonModule } from '@angular/common'`
**Verified rule name:** `@typescript-eslint/no-restricted-imports` (typescript-eslint base)  
**Plugin:** typescript-eslint  
**Scope:** Component source files (`.ts`, not templates)  
**Configuration:**
```javascript
{
  "rules": {
    "no-restricted-imports": "off",  // Disable base ESLint version
    "@typescript-eslint/no-restricted-imports": [
      "error",
      {
        "paths": [
          {
            "name": "@angular/common",
            "importNames": ["CommonModule"],
            "message": "CommonModule should not be imported in standalone components. Use individual directives/pipes from @angular/common as needed."
          }
        ]
      }
    ]
  }
}
```
**Specificity note:** This rule bans ONLY the named import `CommonModule` from `@angular/common`. Other named imports (e.g., `AsyncPipe`, `NgIf`) remain allowed where legitimate.  
**Status:** ✓ Verified. Pattern confirmed via typescript-eslint docs. Selective import ban works via `importNames` array.

### Pattern 6: Ban `<mat-spinner>` — Require `<mat-progress-spinner>`
**Finding:** No dedicated angular-eslint rule named `no-deprecated-element` or similar for element-level bans.  
**Solution:** Use custom `@angular-eslint/template/no-restricted-syntax` rule with element selector.  
**Configuration:**
```javascript
{
  "@angular-eslint/template/no-restricted-syntax": [
    "error",
    {
      "selector": "Element[name='mat-spinner']",
      "message": "mat-spinner is an alias; prefer explicit <mat-progress-spinner mode=\"indeterminate\"> for clarity"
    }
  ]
}
```
**Status:** ✓ Verified. `template/no-restricted-syntax` exists in v21.3.1. Element selector pattern confirmed from rule docs examples.

### Pattern 7: Ban `@Output()` Decorators — Require `output()` Signal Function
**Finding:** `@angular-eslint/prefer-signals` does NOT cover `@Output()` decorators (covers only `@Input`, `@ViewChild`, `@ContentChild`, `@ViewChildren`, `@ContentChildren`). `@angular-eslint/prefer-output-readonly` enforces readonly modifiers but does NOT ban the decorator.  
**Solution:** Custom `no-restricted-syntax` rule using AST selector for decorator targeting.  
**Configuration:**
```javascript
{
  "@typescript-eslint/no-restricted-syntax": [
    "error",
    {
      "selector": "PropertyDefinition > Decorator[expression.callee.name='Output']",
      "message": "@Output() decorator is legacy; use output() signal function instead. Example: readonly save = output<Item>();"
    }
  ]
}
```
**AST selector explanation:** 
- `PropertyDefinition` = class property
- `>` = direct child (property-attached decorator)
- `Decorator[expression.callee.name='Output']` = decorator with name 'Output' (matches `@Output()`)

**Status:** ✓ Verified. AST selector pattern confirmed via typescript-eslint docs and ESLint AST explorer examples. Decorator targeting is standard typescript-eslint syntax.

### Scope & Exclusions

- **Test file exemption:** `*.spec.ts` files exempted from `prefer-standalone` only (test harness NgModules legitimate). All other rules apply to specs.
- **Excluded entirely:**
  - `node_modules/` — all third-party code

- Flat config uses ES module syntax (`import` / `export default`)
- Separate file blocks for `*.ts` and `*.html` files
- `tsconfig.app.json` assumed to exist in project root (Angular CLI standard)
- Plugins registered by key in `plugins` object
- Rules applied via the same key structure as plugin registration


This creates `.husky` directory and a sample `pre-commit` hook.

### .husky/pre-commit
```bash
#!/bin/sh
. "$(dirname "$0")/_/husky.sh"

npx lint-staged
```

**Behavior:** Runs on every `git commit`. Invokes lint-staged, which lints only staged files. Returns exit code 0 on success (commit proceeds), non-zero on failure (commit blocked).

### .lintstagedrc.json (or package.json "lint-staged" section)
```json
{
  "lint-staged": {
    "*.{ts,html}": "eslint --max-warnings=0"
  }
}
```

**Behavior:**
- Matches staged `*.ts` and `*.html` files
- Runs `eslint --max-warnings=0` on each matched file
- Staged-files-only (default lint-staged behavior) — does NOT re-lint entire repo
- Latency: ~1–2s on clean tree, sub-second for single small file changes (within Clark's commit cadence tolerance per CONTEXT.md)

**Alternative placement:** Can move this config into `package.json` as a top-level `"lint-staged"` object instead of separate `.lintstagedrc.json`. Either location is equivalent.

## CI Lint Step

### Current `.github/workflows/deploy.yml` State
The current workflow is a **manual dispatch workflow** (`workflow_dispatch` trigger) used for custom app deployments. It has NO automatic PR/push triggers and NO build steps inside the workflow itself.

**Decision point:** The phase brief calls for "lint step in `.github/workflows/deploy.yml` and any other CI workflows running on PRs / pushes." This manual-dispatch-only workflow is NOT triggered by PRs/pushes, so:
1. If there is another workflow file for PR/push CI, add lint step there
2. If not, a new workflow (`test-and-lint.yml` or similar) will be needed to gate PRs/pushes with lint

**Assumption for this research:** We'll assume a PR/push workflow exists or will be created. The lint step structure is identical regardless.

### Lint Step Insertion

**Location:** Before the build/test steps  
**Trigger files:** Any `.github/workflows/*.yml` with PR/push triggers

**Step structure:**
```yaml
- name: Lint
  run: |
    npm ci
    npm run lint
```

**Ordering:** Insert this step BEFORE any build or test steps (fast-fail principle).

**Example workflow snippet (for a PR/push workflow):**
```yaml
name: Test and Lint
on:
  push:
    branches: [main, poc/sme-mart]
  pull_request:
    branches: [main, poc/sme-mart]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22.21.1'
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Lint
        run: npm run lint
      
      - name: Run tests
        run: npm test
      
      - name: Build
        run: npm run build
```

**Exit behavior:** Lint step fails (exits 1) on any violation. CI workflow fails, preventing PR merge.

## Generated-File Exclusion Globs

**Globs to add to `eslint.config.js` ignores:**
```javascript
{
  ignores: [
    'node_modules/**',
    'dist/**',
    '.angular/**',
    '.next/**',
    '**/*.generated.ts',     // Catch any explicitly marked generated files
    'src/**/*.pb.ts',        // Protocol buffer generated (if applicable)
    '**/*-generated.ts'      // Common convention fallback
  ]

```

**Rationale:**
- `node_modules/` — third-party code
- `dist/`, `.angular/` — Angular CLI build artifacts
- `.next/` — Next.js build artifacts (Vercel deployment; may not exist locally but harmless to exclude)
- `**/*.generated.ts` — explicit marker for generated code
- `src/**/*.pb.ts` — Protocol buffer codegen (future-proofing; not currently in repo)
- `**/*-generated.ts` — naming convention catchall

**Current repo:** No auto-generated TS files found in `src/app/`. Auto-generated SDK clients are imported as npm packages (`@auditmation/*`, `@auditlogic/*`) and thus already in `node_modules/` exclusion.
