## Key References for Executors

### zbb Stack Patterns (from RESEARCH.md)

**Pattern 1: Stack Dependency Graph**
- `depends:` in zbb.yaml; zbb resolves + starts in topo-sorted order
- Example: postgres → minio + registry (parallel) → hub-server → cloudfront-sim

**Pattern 2: Sub-stacks for Composition**
- `substacks:` within single manifest for independently-startable units
- Example: `zbb start sme-mart-spa:hub-server` vs `zbb start sme-mart-spa`

**Pattern 3: Env Import/Export Aliasing**
- Import with `as` to avoid name collisions
- Example: `CLOUDFRONT_SIM_PORT as PROXY_PORT`

**Pattern 4: Health Checks as Readiness Gates**
- `health:` defines command + interval + timeout
- Example: `curl -sf http://localhost:${HUB_SERVER_PORT}/health >/dev/null`

### Common Pitfalls (from RESEARCH.md)

| Pitfall | Risk | Mitigation |
|---------|------|-----------|
| Unverified Hub Module Build | Stack depends on broken module; npm install hangs or resolves wrong version | Wave 0 task: `npm run build` + `npm pack` must succeed (DONE) |
| Port Conflicts | Two stacks claim same port; service won't bind | Let zbb allocate ports; no hardcoded ports in compose.yml |
| Env Var Mismatches | Stack imports var dependency doesn't export; var undefined at runtime | `zbb stack add` validates; `zbb env explain VAR` traces resolution |
| cloudfront-sim Misconfiguration | Deep routes return 404 instead of index.html; basePath wrong | Test locally: `curl http://localhost:PORT/sme-mart/rfps/abc123` should return 200 + HTML |
| Verdaccio Publishing Fails | Module not published to local registry; hub-server can't resolve it | Verify module built + Verdaccio running before publish |
| minio Bucket Initialization | SPA assets not uploaded; nginx returns 404 | setup.sh automates bucket creation + asset upload |

### Build Output Path (from angular.json)

- SPA outputPath: **`dist/`** (default, no explicit outputPath set in angular.json)
- Build command: `npm run build` (or `ng build`)
- Assets to upload: Everything in `dist/` directory

   - On file not found (404 from S3), nginx should fall back to index.html (SPA behavior)
   - Deep routes like `/sme-mart/rfps/abc123` should return `/sme-mart/index.html` (not 404)

   - The SPA may be served from a sub-path like `/sme-mart/` (not root)
   - nginx must:
     - Route `/sme-mart/*` to minio bucket root (strip basePath from request)
     - Serve `/sme-mart/index.html` when routing fails
     - Preserve asset paths (CSS/JS) relative to basePath

3. **Proxy to minio S3 backend**
   - Use `proxy_pass` directive to route requests to minio at `${AWS_ENDPOINT}`
   - Set authentication headers for minio (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY) if needed
   - Or rely on minio being publicly accessible (dev environment, not prod)

4. **Logging + debugging**
   - Access logs for requests (helpful for debugging 404s)
   - Error logs for minio connection issues

**nginx.conf outline (example structure):**
```nginx
user nginx;
worker_processes auto;
error_log /var/log/nginx/error.log warn;
pid /var/run/nginx.pid;

events {
    worker_connections 1024;
}

http {
    include /etc/nginx/mime.types;
    default_type application/octet-stream;

    log_format main '$remote_addr - $remote_user [$time_local] "$request" '
                    '$status $body_bytes_sent "$http_referer" '
                    '"$http_user_agent" "$http_x_forwarded_for"';

    access_log /var/log/nginx/access.log main;

    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;
    keepalive_timeout 65;
    types_hash_max_size 2048;

    # Upstream minio backend
    upstream minio {
        server ${AWS_ENDPOINT};  # or parse host:port separately
    }

    server {
        listen 80;
        server_name _;

        # Serve basePath=/sme-mart/
        location /sme-mart/ {
            # Proxy to minio bucket
            proxy_pass http://minio/${MINIO_BUCKET}/;
            
            # Try original request; if 404, fall back to index.html
            error_page 404 =200 /sme-mart/index.html;
            
            # Preserve headers
            proxy_set_header Host $http_host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }

        # Fallback for 404 errors
        location = /sme-mart/index.html {
            proxy_pass http://minio/${MINIO_BUCKET}/index.html;
            proxy_set_header Host $http_host;
        }

        # Health check endpoint (for zbb)
        location /health {
            return 200 "OK";
            add_header Content-Type text/plain;
        }
    }
}
```

**Key constraints:**
- `try_files` directive logic must be: try original file, if 404, return index.html

- basePath `/sme-mart/` must be consistent with Angular build configuration (check environment files)
- `proxy_pass` to minio: must handle hostname/port correctly (minio may be `localhost:9000`, `http://minio:9000`, etc.)
- `${AWS_ENDPOINT}` is env var passed by docker-compose; nginx needs env var substitution (use envsubst or template)
- Health check endpoint `/health` returns 200 (required by zbb health check)

**Env var substitution:** If nginx doesn't natively support env vars in config (it doesn't), use one of:
1. `docker run -e` + `envsubst` entrypoint script to template the config
2. `COPY` a template and `RUN envsubst < template > nginx.conf` in Dockerfile
