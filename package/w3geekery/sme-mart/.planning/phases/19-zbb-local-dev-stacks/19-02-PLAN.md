## Login Stack Design (from RESEARCH.md)

**cloudfront-sim reuse:** The login stack depends on cloudfront-sim but serves different content (login/ build) from a different minio bucket.

**Bucket separation:** 
- SPA stack: `MINIO_BUCKET=sme-mart-app`
- Login stack: `MINIO_BUCKET=sme-mart-login`
- Both served by same nginx reverse proxy (cloudfront-sim), different location blocks or URL prefixes

**Session handoff:** Platform handles auth (ZeroBias), not stack-specific. Session is cookie/token in browser. Verify by:
1. Load login page at `localhost:PORT/auth/`
2. Inspect browser DevTools → Application → Cookies
3. Navigate to SPA; verify auth token/cookie persists
4. Check SPA Network tab → requests include auth header (from ZB SDK)

**Manifest structure:**

- `name: "sme-mart-login"`, `version: "1.0.0"`
- `depends:` cloudfront-sim only (reused from Wave 1)
  - cloudfront-sim: `./cloudfront-sim` (local relative path to sibling stack)
- `exports:` `[LOGIN_URL, CLOUDFRONT_SIM_URL]`
  - LOGIN_URL is an alias/computed value pointing to the login app on cloudfront-sim
  - Example: `LOGIN_URL: "http://localhost:${CLOUDFRONT_SIM_PORT}/auth/"`
- `imports:` from cloudfront-sim
  - CLOUDFRONT_SIM_PORT, CLOUDFRONT_SIM_URL, AWS_ENDPOINT, AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY (all passed to minio)
- `env:` schema
  - LOGIN_BUILD_DIR: `{ type: string, default: "dist/", description: "Login repo build output" }` (check actual path in login/package.json)
  - MINIO_BUCKET: `{ type: string, default: "sme-mart-login", description: "MinIO bucket for login assets" }` (different from SPA bucket)
  - LOGIN_URL: computed from CLOUDFRONT_SIM_PORT
  - All imported cloudfront-sim vars
- `substacks:` (optional; can be single substack or no substacks)
  - `login-upload`: services=[login-upload], depends=[minio] (one-shot container)
- `lifecycle:`
  - `build:` builds login/ repo (`cd /path/to/login && npm run build`)
  - `start:` calls `bash setup.sh` then `docker compose -f docker-compose.yml -p ${STACK_NAME} up -d`
  - `health:` checks login page accessible via cloudfront-sim
  - `cleanup:` removes containers, images, minio bucket

**Key constraint:** cloudfront-sim is a **dependency**, not a sub-stack. The login stack starts after cloudfront-sim is healthy. This is different from sme-mart-spa which had cloudfront-sim as a composed sub-stack.

Actually, reviewing the architecture: cloudfront-sim could be either:
1. A separate sub-stack in both sme-mart-spa and sme-mart-login (so it's shared + deduped by zbb)
2. Part of sme-mart-spa + reused by importing its exports

**Simplest approach:** cloudfront-sim is a **shared dependency stack**. Both sme-mart-spa and sme-mart-login declare it in `depends:`. zbb starts it once, both stacks import from it.

**Build output path for login/:** Check the login repo's `package.json` build script and output directory. If not found, assume `dist/` (standard Metalsmith/Handlebars output).

**Example manifest outline:**
```yaml
name: "sme-mart-login"
version: "1.0.0"

depends:
  cloudfront-sim:
    package: "./cloudfront-sim"
    ready_when:
      status: healthy

exports:
  - LOGIN_URL
  - CLOUDFRONT_SIM_URL

imports:
  cloudfront-sim:
    - CLOUDFRONT_SIM_PORT
    - CLOUDFRONT_SIM_URL
    - AWS_ENDPOINT
    - AWS_ACCESS_KEY_ID
    - AWS_SECRET_ACCESS_KEY

env:
  LOGIN_BUILD_DIR:
    type: string
    default: "dist/"
    description: "Login repo build output directory"

  MINIO_BUCKET:
    type: string
    default: "sme-mart-login"
    description: "MinIO bucket for login assets (separate from SPA bucket)"

  LOGIN_URL:
    type: string
    value: "http://localhost:${CLOUDFRONT_SIM_PORT}/auth/"
    description: "Public URL for login page"

  CLOUDFRONT_SIM_PORT:
    type: port
    description: "Imported from cloudfront-sim"

  CLOUDFRONT_SIM_URL:
    type: string
    description: "Imported from cloudfront-sim"

  AWS_ENDPOINT:
    type: string
    description: "Imported from cloudfront-sim"

  AWS_ACCESS_KEY_ID:
    type: string
    description: "Imported from cloudfront-sim"

  AWS_SECRET_ACCESS_KEY:
    type: secret
    description: "Imported from cloudfront-sim"

lifecycle:
  build: |
    cd /path/to/login && npm run build
  start: |
    bash setup.sh
    docker compose -f docker-compose.yml -p ${STACK_NAME} up -d
  health:
    command: "curl -sf http://localhost:${CLOUDFRONT_SIM_PORT}/auth/ >/dev/null"
    interval: 3
    timeout: 30
  cleanup:
    - docker compose down -v
    - aws s3 rm s3://${MINIO_BUCKET} --recursive --endpoint-url ${AWS_ENDPOINT} 2>/dev/null || true
```

**Key differences from sme-mart-spa:**
- No postgres, registry, hub-server dependencies (simpler)
- No substacks (could have one for clarity, but optional)
- Depends on cloudfront-sim as external stack (not composed within)
- Login-specific bucket name + build path
- Health check pings login URL, not hub-server

- Command: Bash script that:
  1. Waits for cloudfront-sim to be healthy
  2. Builds login repo if not already built

**Alternative service structure:**
If login/ repo requires Node.js + npm to build (not just static assets), use:
- Image: `node:20-alpine` (includes Node.js + npm)
- Command: build login, then upload

**Example compose.yml:**
```yaml
version: "3.9"
services:
  login-upload:
    image: node:20-alpine
    volumes:
      - /path/to/login:/login:ro
      - ./upload.sh:/upload.sh:ro
    environment:
      AWS_ENDPOINT: ${AWS_ENDPOINT}
      AWS_ACCESS_KEY_ID: ${AWS_ACCESS_KEY_ID}
      AWS_SECRET_ACCESS_KEY: ${AWS_SECRET_ACCESS_KEY}
      MINIO_BUCKET: ${MINIO_BUCKET}
      LOGIN_BUILD_DIR: ${LOGIN_BUILD_DIR}
    command: ["/bin/sh", "/upload.sh"]
```

**Or simpler:** Inline command as multi-line script.

**Key constraint:** No hardcoded port numbers, paths. All use ${VAR} substitution.

**Volume mounts:** `/path/to/login` MUST be an absolute path, or docker-compose will treat it relative to the compose file directory. Use `${LOGIN_REPO_PATH}` env var if variable path needed.

**Depends on:** cloudfront-sim (soft dependency; both are on same docker network)

   - Stop stacks: `zbb stop sme-mart-spa`, `zbb stop sme-mart-login`
   - Cleanup: `zbb slot clean local` (removes containers, images, volumes)
   - Or leave running for next session (Docker persists state)

9. **Troubleshooting**
   - **Port conflicts:** `zbb status` shows allocated ports; if conflict, create new slot with different range
   - **Hub module not found:** Verify Verdaccio is running (part of minio stack), verify module was published
   - **SPA returns 404 on deep routes:** Check nginx try_files config in cloudfront-sim/nginx.conf
   - **Assets not loaded:** Check minio bucket + assets were uploaded; run `aws s3 ls` or check docker logs
   - **Hub-server fails to start:** Check postgres + registry are healthy first; check module version in hub-server env var
