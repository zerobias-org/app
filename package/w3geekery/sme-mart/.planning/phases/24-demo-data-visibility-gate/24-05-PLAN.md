Content should include:

**1. Visibility Gate Effectiveness (Option X — client-side post-filter):**
The demo-visibility gate (DG-02/DG-03) is applied on all list/search results returned from user-facing services after Phase 24 executes. The gate is a CLIENT-SIDE post-filter (`DemoVisibilityService.applyVisibility(records)`) that drops records carrying either demo UUID. Records with `tag: null`, `tag: []`, or non-demo tags pass through unchanged. Server-side `tag` negation is non-viable per Decision-Probe-1 (FAIL 2026-05-01) and is NOT used.

**2. Pre-existing Demo Records (Retroactive Push):**
Existing demo records on UAT that were seeded BEFORE Phase 24 do NOT have the `Object.tag` field populated — they have `tag: null`. **Under Option X, `tag: null` records PASS the post-filter (visible to all users)** because the predicate `isLocalDemoTagged()` returns `false` for null/empty tag arrays. This is by design: legitimate non-demo records also commonly have `tag: null`, and we cannot distinguish them from pre-Phase-24 demo records without retroactively tagging.

**Example:**
```
Pre-Phase-24 demo record:
- id: 'eng-001'
- name: 'Crystal Harbor'
- tag: null  <- No Object.tag field
- isLocalDemoTagged(record) -> false
- Result: Record PASSES the gate (visible to non-admin users)
```

**3. Solution: Retroactive Re-Push (Separate Director Brief)**
A separate director-authored brief (NOT a Phase 24 plan) will re-push all existing demo records via Pipeline.receive with the tag field populated.

Example brief location: `.planning/director/backlog/retroactive-demo-tag-push.md` (TBD)

Example batch: Re-push Engagement, Bid, Note, etc. fixtures with `tag: [{ value: DEMO_TAG_UUIDS.GLOBAL_DEMO }]`.

**4. Timeline:**
- Phase 24 ships: visibility filter infrastructure in place, new demo records tagged at ingest
- Retroactive push (separate brief): existing records re-tagged (optional, can happen immediately or deferred)
- Post-retroactive-push: ALL demo records (old + new) filtered correctly

**5. Until Retroactive Push:**
Non-admin users STILL see pre-existing demo records that lack the `Object.tag` field. This is expected. The gate is EFFECTIVE for all new demo data seeded after Phase 24 executes.

**6. Testing Implication:**
- Phase 24 tests verify filter on fixtures (which include tag field)
- Real UAT may show inconsistency (some demo records visible, some not) until retroactive push runs
- This is NOT a bug; it's expected interim state

**7. Implementation Details:**
- Seeder files (demo-data-seeder.ts, scripts/demo/helpers.ts) updated in Wave 1 to include tag field on all fixtures
- Any NEW seeding (e.g., via demo:seed script) creates records with the tag field
- Existing records are untouched by Phase 24; only retroactive push affects them

Recommendation: Run the retroactive push brief on UAT immediately after Phase 24 executes to ensure demo data is fully gated. Brief scope: query all demo records by `engagementTag = "demo-seed"` (or whatever discriminator exists), re-push via Pipeline.receive with the tag field, verify tag field is now populated on all records.

**Owner:** Director (brief authoring) + optional Phase 24 executor (can run the brief immediately on UAT if desired).
