   a. **Query for demo-tagged class-Objects** (per className listed in researcher inventory — Engagement, SmeMartProject, Bid, BidResponse, Review, Note, NoteFolder, Document, ServiceOffering, etc.):


<threat_model>
## Security Considerations

| Threat | Category | Component | Disposition | Mitigation |
|--------|----------|-----------|-------------|-----------|
| Unauthorized demo-data deletion | Authorization (Elevation of Privilege) | /admin/demo-data route | Mitigate | Route is under /admin, which is protected by onboardingGuard. Only users with `projectContext.isAdmin() === true` reach /admin. Non-admin users cannot access the delete UI. Mitigation: onboardingGuard enforces admin check at route entry. |
| One-click accidental deletion | User Error | Delete button | Mitigate | MatDialog confirmation gate required before deletion proceeds. No one-click trigger. User must explicitly click "Delete" button in dialog. Mitigation: Confirmation pattern is mandatory per CONTEXT.md. |
| Deletion via API request spoofing | Authorization | Pipeline.deleteEntities / hydra delete | Mitigate | Admin check happens at route entry (onboardingGuard). Service does NOT re-check admin status on every deletion call. If a user reaches the service code somehow, they can trigger deletion. Mitigation: Assume route-level guard is sufficient (onboardingGuard enforces admin at /admin route entry). Additional defensive check: service could re-verify admin before deletion (optional enhancement). |
| Bulk deletion without per-item confirmation | User Error | Bulk delete workflow | Accept | Bulk deletion happens with a single confirmation dialog (not per-record). This is acceptable for a deliberate admin action. Rationale: demo data is intentional cleanup, not user-generated content. Non-destructive alternative (soft-delete with markDeleted flag) is already being used (not permanent removal). |
| Error information disclosure | Information Disclosure | MatSnackBar error toast | Mitigate | Error messages show user-friendly text ("Failed to delete Engagement: <error>") not raw stack traces. Backend error details are NOT exposed to the UI. Mitigation: Service catches errors and wraps them in human-readable messages before showing to user. |

## Risk Assessment

- **Medium:** Unauthorized access to /admin route (mitigated by onboardingGuard at route entry)
- **Low:** Accidental one-click deletion (mitigated by MatDialog confirmation)
- **Low:** Information disclosure via error messages (mitigated by error wrapping)
- **Deferred to architecture:** Re-verification of admin status on every service call (optional defensive layer, not in scope)

</threat_model>
