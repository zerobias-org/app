## Strategy — Option X (client-side post-filter, LOCKED)

**Server-side filtering is dead.** Decision-Probe-1 (2026-05-01) verified both `.ne.` and `.not in.` are silently broken on tag arrays in ZB GQL — they return empty regardless of actual tag values. Plus SQL NULL semantics would have excluded `tag: null` records (22/25 in W3Geekery boundary today) that must remain visible. See `24-CONTEXT.md` "Decision-Probe-1 RESULT" for evidence.

**Phase 24 wires the gate as a client-side post-filter.** Each service:
1. Injects `DemoVisibilityService` via field-level `inject()`.
2. After receiving fetched records from `GraphqlReadService.query()`, wraps them with `demoVisibility.applyVisibility(records)` before returning to callers.
3. The service's existing GQL filter logic is **unchanged** — no `tag` filter is added at the GQL layer. The post-filter is the gate.

**Pagination caveat:** the post-filter runs after `pageSize` records have already been fetched. If 3 of 25 returned records are demo, the user sees 22, not a back-filled 25. v1.4 dataset is < 100 records per class — under-fill is cosmetic and acceptable. **DO NOT implement compensating over-fetch logic.** If pagination accuracy becomes user-visible, escalate to v1.5 (Option Y migration) rather than papering over it.

**Alternative considered, REJECTED:** GraphqlReadService augmentation with an `excludeTagUuids?: string[]` parameter would still require working server-side negation. Dead path. Per-service post-filter is the only viable surface.

| Threat | Category | Component | Disposition | Mitigation |
|--------|----------|-----------|-------------|-----------|
| Client-side gate is bypassable in principle (curl / DevTools / tampered SPA) | Authorization | DemoVisibilityService.applyVisibility | **Accept (with caveats)** | The visibility gate is application-layer. A determined user with API access can fetch unfiltered records directly via GQL or curl — the post-filter only protects what the SPA renders. **This is acceptable for Phase 24** because: (1) demo records on UAT are not sensitive (test fixtures, no PII / no real customer data); (2) the threat actor scenario (legit org member with API token willing to bypass UI to see demo data) is implausible and low-impact; (3) Phase 24's stated goal is *visibility hygiene* in the SPA, not authorization. **Document the limitation** in `24-SUMMARY.md`. If the threat model evolves (real PII tagged as demo, untrusted org members), escalate to Option Y (positive include-tag with backfill) where the server-side `.eq.<marketplace>` filter enforces the gate at the API layer. |
| Missing post-filter on one service | Authorization | Service-by-service `applyVisibility()` wrap | Mitigate | If one of 21 services skips the wrap, non-admin users see demo records via that service.
