For each of the 60 call sites listed in `.planning/director/phase-20-call-site-audit.md`:

1. **Read the call site:** Open the file at the listed line number and read the surrounding context (the service method that contains `pushEntity` or `pushEntities`).

2. **Identify the calling component:** Trace back to the component or route handler that invokes the service method. Read enough of that component to answer:
   - Is this a user-initiated action (form submit, button click) or background work (activity log, demo seeding)?
   - If fire-and-forget, does the caller have ANY error surface today? (toast, inline form error, disabled button feedback, retry prompt)
   - If awaited, does the caller actually surface the thrown error to the user, or does it swallow it?

3. **Refine criticality:**
   - **CRITICAL:** User-initiated action that fails silently (no error surface). Loss of user expectations (user thinks "saved", didn't). Includes:
     - Fire-and-forget with NO error surface (silent loss)
     - Awaited call sites whose caller swallows the throw (verified by reading the caller)
   - **MEDIUM:** Collaboration/consistency loss. Failure is eventually visible (missing note on reload, folder didn't move) but user doesn't know immediately. Fire-and-forget only (awaited sites get fixed before wave close).
   - **LOW:** Telemetry/activity. Fire-and-forget for logging/analytics. Idempotent; a missed row doesn't corrupt state.
   - **AWAITED-VERIFY:** Already uses `await`. Phase 1 verifies the caller surfaces error to user; if it doesn't, promote to CRITICAL+SIMPLE for Wave 2.

4. **Refine remediation complexity** (only for CRITICAL and MEDIUM sites):
   - **SIMPLE:** Replace `.catch(console.error)` with `await` + `MatSnackBar.open()` toast. No form state changes needed. Examples: notes, folders, admin workflows.
   - **MEDIUM:** `await` + new form error state (disable submit button, inline red error text, dirty flag). Caller component needs UX updates. Examples: bids, profile saves, service offerings.
   - **COMPLEX:** `await` + retry queue, optimistic-UI rollback, or batch semantics. Requires UX re-architecture. Example: `bids.service.ts:368` (batch bid submit), `vetting.service.ts:184` (batch vetting).
   - **N/A:** For LOW/AWAITED sites, no remediation complexity rating (defer to Wave 2 decision).
