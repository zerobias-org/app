## Remediation Pattern (SIMPLE)

Standard pattern for all SIMPLE remediation call sites:

**Before (fire-and-forget):**
```typescript
this.pipelineWrite.pushEntity(className, data).catch(err => console.error(err));
```

**After (await + toast):**
```typescript
try {
  await this.pipelineWrite.pushEntity(className, data, 'call-site-identifier');
  // Optional: success toast or silent success
} catch (err) {
  this.snackBar.open(`Failed to save ${entityName}: ${err.message}`, 'Dismiss', { duration: 5000 });
  throw err; // Let caller decide (usually re-enable submit button)
}
```

Or for services that don't have direct snackBar access, surface error to caller via thrown exception (caller (component) is responsible for toast).

**Test pattern (TDD):**
1. Write failing test: assert that on rejection, MatSnackBar.open() is called with error message (or exception is thrown).
2. Implement await + surface.
3. Verify test passes.
4. Mock shape derived from real SDK (Pipeline.receive rejection, HttpError, etc.) per `feedback_tests_passing_against_wrong_shape_mocks.md`.

### Service method ends with fire-and-forget `.catch(console.error)` on user-triggered action — BLOCK

**Pattern:** `someMethod() { ...; this.pipelineWrite.pushEntity(...).catch(err => console.error(err)); }`

**Why:** Fire-and-forget masks validation/schema failures from users. If a user-initiated action (form submit, button click) calls a method that fires-and-forgets, the user sees no error if the write fails.

**Fix:** Replace with `await` + user-visible error surface (toast or form error).

**Exception:** Only legitimate for telemetry/activity-log writes (LOW criticality, idempotent). All CRITICAL user actions must use `await`.

**Checker:** If submitting a feature with a user-triggered action + `pushEntity` call, must verify call is `await`ed AND error is surfaced to user.
