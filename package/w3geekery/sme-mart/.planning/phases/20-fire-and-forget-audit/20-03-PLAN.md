## E2E Verification Pattern

**Scope:** Every remediated call site (all CRITICAL+SIMPLE sites from Wave 2) needs at least one test that:
1. Triggers the user action (click button, submit form, etc.) — simulated in the component spec
2. Mocks PipelineWriteService to reject
3. Asserts the user-visible error surface fires (MatSnackBar.open, inline form error, button re-enabled, etc.)

**Test location:**
- Service-level specs (already written in Wave 2): verify service throws on rejection
- Component-level specs (NEW in Wave 3): verify component catches service error and surfaces to user (toast, form state, etc.)

**Benefit:** Confirms the whole chain works, not just the service in isolation.

## Round-Trip Verification Approach

**For each of 23 class IDs in SME_MART_CLASS_IDS:**

1. Construct minimal-but-valid data for that class (derive from schema or demo-data helpers)
2. Call `pushEntities(className, [data])` via PipelineWriteService
3. Immediately query GraphQL: `ClassName(id: "...") { ... }`
4. Confirm the row was persisted (not rejected by validation, not lost due to fictional class ID)
5. Log result: ✅ (success) or ❌ (failure, with reason)
6. If any ❌ and reason is "class ID mismatch" or "no such class", file errata immediately

**Environment:** Use UAT (not prod). Authenticate with ZB MCP using the UAT profile.

**Cleanup:** After round-trip testing, delete the test rows via GQL mutation or `markDeleted` via Pipeline (optional, depends on test data cleanup policy).

## UAT Soak Configuration

**Duration:** 1 week post-merge (7 days)
**Environment:** UAT (`uat.zerobias.com`)
**Measurement:** Telemetry rejection events for the 44 fire-and-forget sites

**Soak hypothesis:**
- Before Wave 2: any failure in a CRITICAL+SIMPLE site produces silent console.error, user sees nothing
- After Wave 2: any failure produces visible error + telemetry event
- Soak observes: do any of the 44 fire-and-forget sites actually reject? If non-zero rejection events fire in a full week of normal UAT usage, that confirms errata 011's hypothesis is still valid and future work (Waves 2-3) is justified

**Soak success criteria:**
- Telemetry sink is queryable (CloudWatch logs / Sentry dashboard / console parse)
- Events are tagged with className, callSite, timestamp
- For the 1-week observation period, rejection count per site is available

**Note:** Soak does NOT block phase close. Post-merge, soak runs independently; Director reviews soak report 1 week later to decide if any hot-fix is needed (e.g., if a critical site was rejecting 100x/day, prioritize that remediation in v1.5).
