## RFP Access Controls (D1)

**D1-01 entity model:** "Boolean flag on project entity" — is this on SmeMartProject directly, or should RFP-specific fields live on a separate entity/section? Today an RFP is a SmeMartProject with `projectType: 'rfp'`. Adding `isInvitationOnly` directly to SmeMartProject means all project types carry that field. Acceptable? Or should we introduce an RFP-specific config section (JSON blob like vendor profile `data` field)?

## Document Packages (D2)

**D2-04 variable substitution scope:** "Buyer name, dates, engagement ID auto-filled" could range from simple find-and-replace (`{{buyer_name}}`) to a full template engine. Recommend scoping to: predefined variables only (no user-defined), simple string replacement, no conditional blocks. Confirm during discuss-phase or mark as stretch goal.

## Form Builder (D3)

**D3-02 wording:** Requirement text includes implementation guidance ("ngx-formly's JSON-based config pattern as reference, not dependency"). The discuss-phase should focus on behavior, not implementation — implementation details belong in the PLAN. Minor, not blocking.

## Pilot Projects (PLT)

**PLT-02 completion trigger:** "Pilot completion creates a conditional vetting checklist item" — what triggers completion? Options: (a) buyer manually marks pilot complete, (b) status field change to `completed`, (c) all pilot tasks resolved. Needs clarification during discuss-phase.

## Cross-Cutting: RFP Status Lifecycle

**Not currently a requirement.** Today RFPs presumably have draft/published/closed states (via SmeMartProject status). Invitation-only adds questions:
- Can buyer invite vendors after publishing?
- Can buyer revoke invitations?
- Does closing an RFP auto-decline pending invitations?
- Can a vendor bid after the RFP is closed?

Either add explicit requirements or mark as out-of-scope for v1.2 during discuss-phase.
