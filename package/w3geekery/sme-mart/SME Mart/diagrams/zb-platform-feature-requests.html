🏗 Brian's Architecture

Brian defined a 4-level hierarchy for how SME Mart engagements work. The ZeroBias platform currently has Tasks (with subtasks) and Boundaries, but is missing the Project layer in the middle. Additionally, the Transparency Center (3-view buyer/supplier/shared) requires subtask typing and visibility controls that don't exist yet.
Engagement (org-to-org: W-9, tax, DUNS, banking, exec background checks) ├── Project (scope of work: where $ happens, specific deliverables) │ ├── Task (how outcomes delivered: apps, services, agents) │ │ ├── SubTask: Legal requirements satisfied │ │ ├── SubTask: $ charged per pricing model │ │ └── SubTask: Functional deliverable │ └── Task 2 ... └── Project 2 ...
Transparency Center aggregates $, compliance, functional, and legal data from SubTask → Task → Project → Engagement — with 3 views: Buyer (requirements), Supplier (how they satisfy), Shared (both parties).

🎯 Platform Feature Requests

FR-001 Project Entity
High Blocking

Add a Project entity that groups Tasks under a named scope within a Boundary. Projects are "where $ happens" — the specific scopes of work within an engagement.

Requirements
  • Projects belong to a Boundary (or Tag-scoped engagement)
  • Multiple projects per boundary
  • Metadata: title, description, status lifecycle, budget/pricing
  • Tasks assigned to projects (many-to-one)
  • Roll-up views: task count, status summary, completion %
  • API: CRUD projects, list by boundary, assign tasks
SME Mart Workaround
Model projects in SME Mart's Neon database initially, linked to ZB Tasks by ID. Migration path: replace Neon project table with ZB Project API calls when platform adds the entity.

Who Else Needs This
Work Worlds also needs project-level grouping for missions within engagements.
FR-002 Structured SubTask Types
Medium

Support typed subtasks — specifically, subtasks categorized as Legal, Financial ($), and Functional. The Transparency Center needs to aggregate these dimensions from subtask level upward.

Requirements
  • SubTask type field: 'legal', 'financial', 'functional', or custom
  • Query subtasks by type within a task
  • Type-specific metadata (e.g., financial → amount field)
  • Roll-up queries: "all financial subtasks across tasks in a project"
Path Forward
ZB Task Activities define custom fields on tasks. Currently not customizable, but near-term plan to allow creating custom Activities — this will solve subtask typing. Question: timeline for custom Activity creation?
FR-003 Task/Boundary Attachment Support
Solved

File attachment support on Tasks for evidence documents, deliverables, and legal documents.

Resolution
SDK Task attachments → AWS. The ZB SDK already supports task attachments uploaded to AWS, owned by ZeroBias. SME Mart will use this existing capability for Transparency Center evidence documents.
No Platform Change Needed
Existing Task attachment API in the SDK is sufficient. SME Mart just needs to integrate it.
FR-004 Boundary-Scoped Data Visibility
Solved

Data visibility rules within a Boundary — resources NOT in the boundary are not visible.

Resolution
ZB Boundary Manager app. Boundaries contain lists of frameworks, products, etc. that must be included for a resource to be utilized. Any resource NOT in the boundary is not visible. This is managed via the ZB Platform Boundary Manager app.
No Platform Change Needed
Existing Boundary scoping handles visibility enforcement. SME Mart consumes boundary membership to determine what each party sees.

SME Mart's Workaround Strategy
Approach: Build with Neon database workarounds now, design for easy migration when platform features arrive.

FR-001 (Projects): projects and project_tasks tables in Neon. Projects reference engagement IDs. Tasks linked by ZB Task ID. When ZB adds Projects, swap Neon CRUD for ZB API calls.

FR-002 (SubTask Types): Use ZB Task tags to categorize subtasks. Store type-specific metadata (amounts, compliance status) in Neon tables keyed by ZB Task ID.

FR-004 (Visibility): Implement buyer/supplier/shared visibility flags in Neon transparency_requirements and transparency_disclosures tables. Enforce in Angular service layer. Less secure than platform-level but functional for MVP.

Migration principle: Use ZB-compatible IDs everywhere. Don't over-customize the data model. Keep the service interface stable so swapping Neon→ZB API is a backend change, not a UI rewrite.
